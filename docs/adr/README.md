# Architecture Decision Records (ADR)

## ADRとは

Architecture Decision Records (ADR) は、ソフトウェア開発における重要なアーキテクチャ上の意思決定を記録するドキュメントです。

### ADRの目的

- **透明性**: なぜその決定がなされたのかを明確にする
- **履歴**: 意思決定の経緯を記録し、将来の参考にする
- **コンテキスト**: 決定時の状況や制約を残す
- **変更管理**: 意思決定の変更や見直しを追跡する

### ADRの書き方

各ADRは以下の構成で記述します：

1. **タイトル**: 決定の簡潔な要約
2. **ステータス**: 提案中/承認済み/却下/廃止
3. **コンテキスト**: 決定が必要になった背景
4. **決定**: 実際の意思決定内容
5. **結果**: 決定による影響や期待される効果
6. **代替案**: 検討したが採用しなかった選択肢

## ADR作成ガイドライン

### いつADRを書くべきか

- 技術スタックの選定（言語、フレームワーク、ライブラリ）
- アーキテクチャパターンの採用
- データモデルの重要な設計決定
- セキュリティやパフォーマンスに関わる決定
- チーム内で議論が必要な決定

### ADRのテンプレート

```markdown
# ADR-XXX: [タイトル]

## ステータス

[提案中 / 承認済み / 却下 / 廃止]

## コンテキスト

[決定が必要になった背景や問題点を説明]

## 決定

[実際の意思決定内容を明確に記述]

## 結果

### ポジティブな影響
- ...

### ネガティブな影響
- ...

### 中立的な影響
- ...

## 代替案

### 案1: [代替案の名前]
- メリット: ...
- デメリット: ...
- 却下理由: ...

## 参考資料

- [関連ドキュメントやリンク]
```
